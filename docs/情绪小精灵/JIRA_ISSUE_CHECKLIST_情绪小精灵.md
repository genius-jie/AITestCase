## Issue依赖关系管理

### 依赖关系类型

1. **强依赖**：必须先完成前置Issue才能开始后续Issue
2. **弱依赖**：建议先完成前置Issue，但可以并行开发
3. **阻塞依赖**：前置Issue阻塞了后续Issue的进度

### 依赖关系示例

**Epic 1内部依赖**
- Story 1.1 - 1.5 → Story 1.6（强依赖）
  - 理由：必须先完成5种人格的实现，才能实现人格自动切换

**Epic 2内部依赖**
- Story 2.1 → Story 2.3（弱依赖）
  - 理由：建议先完成唤醒/待机功能，再实现睡眠模式，但可以并行开发

**跨Epic依赖**
- Story 1.6 → Story 2.1（弱依赖）
  - 理由：建议先完成人格自动切换，再实现语音唤醒，但可以并行开发

### 依赖关系管理最佳实践

1. **明确依赖关系**：在Issue描述中明确标注依赖关系
2. **使用Jira的依赖关系功能**：使用Jira的"链接"功能建立Issue之间的依赖关系
3. **定期检查依赖关系**：在Sprint Planning时检查依赖关系，确保Sprint Backlog的Issue没有未解决的依赖
4. **及时更新依赖关系**：当依赖关系发生变化时，及时更新Jira中的依赖关系

## Issue状态流转

### Story状态流转

```
To Do → In Progress → In Review → Done
```

**状态说明**
- **To Do**：待办状态，Issue已创建但未开始
- **In Progress**：进行中状态，Issue正在开发中
- **In Review**：评审中状态，Issue已完成开发，等待Code Review和测试
- **Done**：已完成状态，Issue已通过测试并合并到主分支

### Bug状态流转

```
New → In Progress → In Review → Done
```

**状态说明**
- **New**：新建状态，Bug已创建但未开始修复
- **In Progress**：进行中状态，Bug正在修复中
- **In Review**：评审中状态，Bug已修复，等待验证
- **Done**：已完成状态，Bug已修复并验证通过

### 状态流转规则

1. **To Do → In Progress**：开发人员开始开发时，将Issue状态从To Do改为In Progress
2. **In Progress → In Review**：开发人员完成开发后，将Issue状态从In Progress改为In Review
3. **In Review → Done**：测试人员验证通过后，将Issue状态从In Review改为Done
4. **In Review → In Progress**：测试人员验证失败后，将Issue状态从In Review改为In Progress，开发人员继续修复

## Issue优先级定义

### 优先级级别

1. **Critical**：严重问题，必须立即修复
2. **High**：高优先级，尽快修复
3. **Medium**：中优先级，按计划修复
4. **Low**：低优先级，有时间再修复

### 优先级判断标准

**Critical**
- 系统崩溃或无法使用
- 数据丢失或损坏
- 安全漏洞
- 核心功能完全不可用

**High**
- 核心功能部分不可用
- 严重影响用户体验
- 性能严重下降
- 主要功能有bug

**Medium**
- 次要功能有bug
- 用户体验一般
- 性能一般下降
- 非核心功能不可用

**Low**
- 界面小问题
- 文档错误
- 非功能性问题
- 优化建议

## Issue创建最佳实践

### 1. Epic创建最佳实践

- Epic名称应该简洁明确，能够概括整个Epic的内容
- Epic描述应该详细说明Epic的目标、范围和验收标准
- Epic应该包含多个Story，每个Story都是独立的可交付成果
- Epic的Story Points应该是所有Story Points的总和
- Epic应该有明确的优先级和版本规划

### 2. Story创建最佳实践

- 用户故事应该遵循"作为[角色]，我想要[功能]，以便[价值]"的格式
- 验收标准应该是可测试、可验证的
- 业务规则应该清晰明确，避免歧义
- Story Points应该基于相对复杂度估算，而不是绝对时间
- Story应该在一个Sprint内完成，避免跨Sprint

### 3. Task创建最佳实践

- Task描述应该清晰明确，能够指导开发人员完成工作
- 验收标准应该是可测试、可验证的
- 技术要求应该明确，避免歧义
- Task应该在1-3天内完成，避免过大的Task
- Task应该有明确的负责人和预估时间

### 4. Bug创建最佳实践

- Bug描述应该清晰明确，能够快速定位问题
- 复现步骤应该详细完整，能够准确复现问题
- 期望行为和实际行为应该对比明确
- 环境信息应该完整，包括操作系统、设备型号、软件版本等
- Bug应该有明确的优先级和严重程度

### 5. Sprint规划最佳实践

- Sprint Backlog的Story Points应该与团队容量匹配
- Sprint应该有明确的目标和交付物
- Sprint应该优先完成高优先级的Story
- Sprint应该预留20%的时间用于处理突发问题
- Sprint应该定期回顾和改进

## 附录：Issue创建快捷命令

### Jira CLI命令示例

```bash
# 创建Epic
jira epic create --summary "小精灵人设和语言风格系统" --description "实现5种情绪人格" --priority High

jira issue create --type Story --summary "用户可以体验快乐人格的陪伴" --description "作为用户，我想要体验快乐人格的陪伴" --priority High --story-points 8

jira issue create --type Task --summary "实现快乐人格的语言风格库" --description "实现快乐人格的语言风格库" --priority High --estimate 8h

# 创建Bug
jira issue create --type Bug --summary "快乐人格在用户低落时回应不正确" --description "当用户处于低落情绪状态时，快乐人格的回应不符合预期" --priority High --severity Major
```
